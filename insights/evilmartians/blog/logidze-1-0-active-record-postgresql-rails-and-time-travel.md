# Logidze 1.0: Active Record, PostgreSQL, Rails, and Time Travel

Logidze 1.0 delivers PostgreSQL-trigger-based record versioning for ActiveRecord — storing a complete change log as JSONB in a `log_data` column on each model table, with time-travel query APIs and no additional tables.

## What Logidze Does

- Tracks every change to an ActiveRecord model's attributes as a PostgreSQL-level trigger — works regardless of update source (Rails, raw SQL, migrations)
- Stores the full change log in a `log_data` JSONB column on the model's own table — no separate audit or versions table
- Enables "time travel": retrieve the exact state of any record at any past timestamp
- Generates diffs between versions: what changed between two timestamps
- Supports `responsible_id` meta-field: attach the ID of the user who made a change to the log entry
- Works with PostgreSQL 9.6+ and Rails 5.2+ (1.0 requires Rails 6+)

## How It Works Internally

- A PostgreSQL PL/pgSQL trigger fires on every `INSERT` and `UPDATE` — the trigger function appends to `log_data`
- `log_data` JSONB structure: `{ v: 2, h: [{ v: 1, ts: 123456, c: { field: 'old_val' } }, ...] }` where `h` is history array
- Each history entry `c` stores only the diff (previous values of changed columns) — not the full row snapshot
- `hstore_to_jsonb_loose` converts `hstore` diffs to JSONB internally — benchmarked as the fastest diffing approach
- The trigger is installed per-table via a Rails migration generated by `logidze:model` generator
- Trigger function is a single shared PL/pgSQL function (`logidze_logger`) called by per-table triggers

## Installation and Setup

- Add `gem 'logidze'` to Gemfile and run `bundle install`
- `rails g logidze:install` — generates two migrations: one creating the shared `logidze_logger` function, one adding `hstore` extension
- `rails g logidze:model ModelName` — generates a migration adding `log_data` column and per-table trigger to a specific model
- Run `rails db:migrate` — the trigger is now active and starts recording changes immediately
- Include in model: `has_logidze` — adds all Logidze instance and class methods to the model
- Existing records get `log_data: null` — first update populates the log; use `logidze:model --backfill` to populate history for existing rows

## Time Travel API

- `record.at(time: 1.day.ago)` — returns a new instance of the model reflecting the record's state at that time
- `record.at!(time: ...)` — raises `Logidze::NotVersionedError` if no log entry exists for that time
- `Model.at(time: 1.week.ago)` — class scope; all queries return records as they were at the given timestamp
- `Model.at(time: ...).where(status: 'active')` — compose with standard ActiveRecord queries
- `record.log_version` — current version number (integer, monotonically increasing)
- `record.log_size` — number of history entries in `log_data`

## Diff and Comparison API

- `record.diff_from(time: 1.day.ago)` — returns a hash of changes made since the given timestamp: `{ 'field' => { 'before' => old, 'after' => current } }`
- `record.diff_from(version: 3)` — diff from a specific version number
- `record.previous_changes` is standard Rails; Logidze's `diff_from` reaches back arbitrarily far
- `record.undo!` — reverts the record to the previous version (triggers a SQL UPDATE via ActiveRecord)
- `record.redo!` — re-applies the last undone change
- `record.switch_to!(version: 2)` — sets the record to an arbitrary past version

## Responsible ID (Who Made the Change)

- Meta-fields can be embedded in each log entry — `responsible_id` is the most common use case
- Set before an update: `Logidze.with_responsible(current_user.id) { record.update!(attrs) }`
- Stored in the `log_data` history entry as `r: <user_id>`
- Retrieve: `record.at(time: ...).log_data.responsible_id` — the ID of who made that version
- Supports arbitrary meta keys: `Logidze.with_meta({ responsible_id: id, reason: 'admin_edit' }) { ... }`
- `Logidze.with_responsible` uses a PostgreSQL session variable (`logidze.responsible_id`) readable by the trigger

## Filtering and Ignoring Columns

- `has_logidze(only: [:status, :title])` — only tracks changes to listed columns
- `has_logidze(except: [:updated_at, :cached_value])` — excludes noisy or derived columns from the log
- Filtering is implemented in the trigger function via PostgreSQL array intersection — efficient, no Ruby overhead
- Combine `only`/`except` with `timestamp_column: :custom_ts` to use a non-standard timestamp column
- Column filtering is set at migration generation time (`--only` / `--except` flags on `logidze:model`)

## Versioning and Migration

- Version 1.0 (2021) introduced: Rails 6 compatibility, full Zeitwerk support, simplified `has_logidze` DSL
- `logidze:model` generator supports `--update` flag to regenerate trigger after changing `only`/`except` options
- `Logidze.without_logging { ... }` block — executes updates without triggering log appends (useful for data migrations)
- The trigger can be temporarily disabled at the session level: `ActiveRecord::Base.connection.execute('SET LOCAL logidze.enabled TO off')`
- `logidze:model --backfill` generates a data migration populating `log_data` for all existing rows from `updated_at` timestamps

## Comparison with Alternatives

- `paper_trail` stores versions in a separate `versions` table with full snapshots — higher storage, simpler querying
- `audited` also uses a separate table with full-row snapshots — similar trade-offs to PaperTrail
- Logidze stores diffs only (not full snapshots) — lower storage, but reconstructing old state requires replaying the diff chain
- Logidze trigger fires at the DB level — captures changes from any source (Sequel, raw SQL, pg jobs); PaperTrail only captures ActiveRecord-mediated changes
- Logidze reads are a single column access (`log_data`) vs. PaperTrail's JOIN to `versions` table — faster point-in-time reads
